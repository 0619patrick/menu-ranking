# 菜单配置表说明

本项目已改为**配置驱动**:每家餐厅的菜单和处理规则不再写在 Python 代码里,
而是放在 `app/menus/data/<餐厅key>/` 目录下的 CSV 表格里。

**改菜单、加 POS 别名、加新店,都只需要编辑表格,不需要改任何代码。**

```
app/menus/data/
├── tiantian/        天天Authentic（香港）
├── tiantian_cn/     天天 Plus⁺（海口/南京/上海）
├── acheng/          阿城鵝肉（香港）
├── acheng_cn/       阿城鹅肉（深圳）
├── taicai/          中環泰菜 Here Thai
└── sijifenfang/     四季芬芳
        ├── menu.csv             ← 菜单主表（最常改的就是它）
        ├── config.csv           ← 其余规则
        └── store_overrides.csv  ← 可选,单店 POS 写法补丁
```

> ⚠️ 编辑须知
> - 文件编码是 **UTF-8(带 BOM)**,用 Excel 双击打开、编辑、保存即可(保存时选「CSV UTF-8」格式)。
> - 多个 POS 写法之间用竖线 **`|`** 分隔,菜名/写法里不要出现 `|` 字符。
> - 改完保存即可,程序**重启后**生效(本地重跑 / Render 重新部署)。

## 一、menu.csv —— 菜单主表

| 列 | 含义 |
|---|---|
| 分类 | 菜单 PDF 上的分类(段标题),相同分类的行要连续放在一起,顺序 = 报表显示顺序 |
| 菜名 | 菜单 PDF 上的标准菜名(报表里显示的名字) |
| 价格 | 数字 |
| 单位 | 份 / 杯 / 碗 …… |
| POS写法 | 这道菜在 POS 里所有可能的名字,多个用 `\|` 分隔 |

### 最常见的三种改法

**1. 某家店 POS 把菜名打得不一样,销量没归进来**
→ 找到那道菜的行,在「POS写法」末尾加 `|新写法`。
例:沙田店把「海南沙律」打成「海南沙拉」:
```
前菜/小食,海南沙律,59,份,海南沙律|海南沙拉
```

**2. 出了新菜**
→ 在对应分类的行块里插入一行,五列填全。

**3. 改价格**
→ 直接改「价格」列。

### 特殊写法
- **空分类占位**:「菜名」留空、只填「分类」的行,表示这个分类的内容
  完全以 POS 实际项目为准(配合 config.csv 的 `pos_native_section`,
  如深圳阿城的「12元精选卤味」)。

## 二、config.csv —— 其余规则

三列固定:`配置项, 键, 值`。同一配置项可以多行。

| 配置项 | 键 | 值 | 说明 |
|---|---|---|---|
| brand | - | 品牌显示名 | 必填,如 `天天Authentic` |
| short_name | - | 短名 | Excel sheet 命名用,如 `天天` |
| drop_category | - | POS 分类名 | 堂食要**丢弃**的 POS 分类(改碼/走/多/少等辅助分类) |
| delivery_platform | 平台名 | 识别标记 | POS「分类」含此标记 = 该平台外卖。一个平台多个标记就写多行 |
| adjust_marker | - | 子串 | POS 分类含它 = 补差价行(如 `手寫單`) |
| pos_alias | POS 名 | 显示名 | 菜单未列但想中文显示的项目(泰菜 Set 套餐) |
| pos_rename | 原 POS 写法 | 统一成的名字 | **统一写法**:错字/字序颠倒/旧名等整体改名,匹配和聚合都用新名。两个错写并成一行时各写一行,值填同一个正确名。例:`pos_rename,鮮無果花燉雞湯,鮮無花果燉雞湯` + `pos_rename,鮮無花果雞湯,鮮無花果燉雞湯` 让两个错写的湯品在「湯品」里合并成一行 |
| addon_category | - | POS 分类名 | 「点选拆分」加料分类(四季芬芳 米線$X餸) |
| addon_section | - | 菜单分类名 | 与 addon_category 配对的菜单段 |
| cat_map | POS 大类 | 菜单分类 / `__OUT__` / `__DROP__` | 未匹配菜单的新菜按 POS 大类归到哪 |
| force_cat | POS 菜名 | 菜单分类 | 按**精确菜名**强制归类(纠正 POS 错放),优先级最高 |
| route_rule | 目标分类 | 条件式 | 按**名字模式**批量归类(force_cat 的批量版),见下方说明 |
| drop_name | - | POS 菜名 | 按菜名直接丢弃(打包盒/纸巾等) |
| main_section | - | 菜单分类名 | 套餐分类下的单品饭/面自动归到此分类 |
| main_keyword | - | 关键字 | 「单品主食」识别字(饭/面/粉…) |
| set_keyword | - | 关键字 | 「套餐」识别字(套餐/+/双人…),命中则进菜單外 |
| pos_native_section | - | 分类名 | 「POS 原生分类」,整段以 POS 实际项目为准、不打🆕 |

例(阿城外卖平台,两个平台、其中 Keeta 有两个标记):
```
delivery_platform,Keeta,KT
delivery_platform,Keeta,Keeta
delivery_platform,Foodpanda,FP
delivery_platform,Foodpanda,【外
```

### route_rule —— 按名字模式批量归类

当一批"未列入菜单的新菜"有**共同命名特征**时,不用一条条 `force_cat` 列菜名,用一条 `route_rule` 全收。
- **键** = 目标菜单分类
- **值** = 条件,用 `;` 分隔,**全部满足才命中**。支持:
  - `cat=X` 只对 POS 大类是 X 的项生效(可选,建议加上以免误伤)
  - `startswith=X` 名字以 X 开头
  - `contains=X` 名字含 X
  - `endswith=X` 名字以 X 结尾
  - 至少要有一个名字条件(startswith/contains/endswith)

例(阿城:POS大类「1人套餐」里、名字以「滷鵝」开头且含「$」的套餐价滷味,全部归「滷味」):
```
route_rule,滷味,cat=1人套餐;startswith=滷鵝;contains=$
```
这样无论哪家店、以后新出现的 `滷鵝X$XX.5` 都会**自动归位**,不用再加菜名。
优先级:精确 `force_cat` > `route_rule` > `cat_map`。

### 其他进阶配置项

| 配置项 | 键 | 值 | 作用 |
|---|---|---|---|
| auto_match_category | - | 1 | 开启「POS大类同名归菜单」:菜單外某 POS 大类名 == 某菜单分类名时,自动把它的项归入该分类(🆕),不必逐个写 cat_map。例:天天的 POS大类「湯品/主食/甜品」自动进菜单同名分类 |
| extras_merge | 源POS大类 | 目标大类 | 把菜單外的「源大类」并进「目标大类」,**同名/同代号**项求和。例:`extras_merge,7點後招牌海南雞飯,午餐` 把7點後并入午餐。目标大类内会按「代号」(名字「. 」前的部分,如午G)聚合,容忍 POS 录入的小差异 |
| strip_token | - | 要忽略的子串 | 匹配前先把这个子串从 POS 名里删掉再对菜单,处理无关后缀。例:`strip_token,,（不能走甜）` 让「凍-檸檬薏米水（不能走甜）」对上菜单的「凍-檸檬薏米水」 |

> 菜單外发生合并的项(extras_merge 或同代号聚合)会自动带蓝色 ▶,点开能看到各来源明细。

## 三、store_overrides.csv —— 单店 POS 写法补丁(可选)

只有「同一品牌下、个别门店写法特殊」时才需要。没有就不用建这个文件。

| 列 | 含义 |
|---|---|
| 店铺 | 前端门店名(QS_STORES 里的 name) |
| 菜名 | menu.csv 里的标准菜名 |
| 补充POS写法 | 该店特有写法,多个用 `\|` 分隔 |

例:
```
店铺,菜名,补充POS写法(多个用|分隔)
香港天天沙田,酥炸椒鹽板豆腐,酥炸椒鹽豆腐
```

## 三点五、同品牌多店、菜单只差一点点 —— 配置继承(extends)

当几家店**菜单基本相同、只有个别差异**时(例:阿城淘大/太古完全一样,東薈城只多一个「手作系列」),
**不要复制整份菜单**。让差异店"继承"主店,只写出自己多出来的部分。

做法:给差异店建一个目录(如 `acheng_donghui/`),`config.csv` 里写:

```
配置项,键,值
extends,,acheng              ← 先继承 acheng 的全部菜单和规则
category_after,手作系列,飲品   ← 新分类「手作系列」插到「飲品」之后
```

它的 `menu.csv` **只写多出来的分类/菜品**:

```
分类,菜名,价格,单位,POS写法(多个用|分隔)
手作系列,芋圓愛玉芒果刨冰,28,份,芋圓愛玉芒果刨冰
手作系列,麻薯豆乳芒果刨冰,28,份,麻薯豆乳芒果刨冰
```

继承规则:
- **菜单**:先拿主店全部分类;`menu.csv` 里**同名分类会覆盖**主店那一类、**新分类**按 `category_after` 插入(没写锚点就放末尾)。
- **规则**(config.csv):丢弃项/外卖标记等与主店**合并**;`brand`/`short_name` 等单值有写才覆盖,没写就用主店的。
- 主店改价/加菜 → 继承的店**自动跟着变**,不用改两处。

> 前端 `index.html` 的 `QS_STORES` 里,把这家店的 `rest` 指向新 key(如 `rest: 'acheng_donghui'`)。

## 四、加一家全新的餐厅(不写代码)

1. 在 `app/menus/data/` 下新建目录,目录名 = 餐厅 key(英文小写,如 `yizheng`)
2. 放入 `menu.csv`(照抄任意一家的表头)和 `config.csv`(至少要有 `brand` 一行)
3. 前端 `index.html` 的 `QS_STORES` 给对应店铺设 `restaurant: 'yizheng'`
4. 重启 / 重新部署,完成

程序启动时自动扫描 `data/` 下所有含 `menu.csv` 的目录完成注册,
不需要在任何代码里"登记"新餐厅。

---

## 五、出错了怎么办

- 启动时报 `加载餐厅配置「xxx」失败: ...`:报错信息里带文件名和行号,
  按提示检查那一行(常见:价格列不是数字、配置项拼错)。
- 改了表没生效:确认程序重启过;确认改的是 `app/menus/data/` 下的文件。
- 历史版本:所有表都在 git 里,`git log -- app/menus/data/` 可查每次改动。
